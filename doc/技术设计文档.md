# hi-im-api 技术设计文档

> **组件**：hi-im-api（Go module + protobuf）  
> **层级**：L2 内部契约（**不独立部署**）  
> **必嗨等价**：`lib/comm` + `lib/mesg` + Thrift seqsvr IDL  
> **版本**：v0.1 · 2026-07-02  
> **生态**：对应 beehive-im 仓库 `doc/hi-im-档C技术方案设计.md` **生态 M2**

---

## 1. 定位与边界

### 1.1 是什么

**hi-im-api** 是全 hi-im Go 栈共享的 **协议与数据契约**：

- IM **48 字节报头** 与 **CMD 命令字**
- **Protobuf** 消息体定义（客户端 WS / 进程间业务帧 payload）
- **Redis 键名** 与格式化模板
- **错误码**、HTTP JSON 响应壳
- **gRPC** `.proto`（seqsvr 发号等冷路径）

各 L3 服务（gateway、usrsvr、msgsvr…）与 **hi-im-hubclient** 通过 `go get` 引用同一版本，避免 cmd/key 漂移。

### 1.2 不是什么

| 不是 | 说明 |
|------|------|
| **独立部署的服务** | 无 main、无 Pod |
| **hi-im-hubclient** | 连 Hub 的 TCP 在 hubclient；api 只定义帧里装什么 |
| **对外 hi-im-sdk-go** | 外部集成方应调 usrsvr HTTP/gRPC，不 import 本库（除非 fork 全栈） |
| **业务逻辑** | 无 fan-out、无 Redis 读写、无 Gin 路由 |

### 1.3 与 hi-im-hubclient 分工

```text
msgsvr 业务代码:
  1. imv1.GroupChatMessage 序列化     ← hi-im-api (proto)
  2. header.Pack(cmd, sid, nid, …)   ← hi-im-api (48B)
  3. hubclient.AsyncSend(cmd, nid, buf) ← hi-im-hubclient (TCP)
```

**变更归属**（详见档 C §4.6）：

| 变更 | 改 api？ | 改 hubclient？ |
|------|----------|----------------|
| 新 cmd / 新 proto | ✅ | ❌ 通常 |
| 新 Redis key | ✅ | ❌ |
| bus wire v2 | ✅（若 IM 头扩展） | ✅ |
| 群 fan-out 算法 | ❌ | ❌（改 msgsvr） |

## 2. 在协议栈中的位置

```text
┌─────────────────────────────────────────────────────────┐
│ 客户端 WebSocket / TCP                                   │
│   [ MesgHeader 48B | protobuf body ]  ← hi-im-api 定义   │
└───────────────────────────┬─────────────────────────────┘
                            │ 接入进程
┌───────────────────────────▼─────────────────────────────┐
│  hi-im-hubclient → hi-im-core (bus wire 20B + 上帧)      │
│   bus wire v1 在 hi-im-core/doc，payload = 整帧 IM 二进制  │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  msgsvr / chatroom 解析 header + proto ← hi-im-api       │
└─────────────────────────────────────────────────────────┘

冷路径：usrsvr ──gRPC(seq.proto)──► hi-im-seqsvr  ← hi-im-api/gen/go/seq/v1
```

**两层头**（勿混）：

| 层 | 长度 | 定义位置 | 用途 |
|----|------|----------|------|
| **bus wire** | 20B | hi-im-core | Hub 路由 cmd/nid |
| **IM MesgHeader** | 48B | **hi-im-api** | 业务 sid/cid/nid/seq/dsid |

## 3. 必嗨迁移对照

| 必嗨 | hi-im-api |
|------|-----------|
| `lib/comm/mesg.go` CMD + MesgHeader + Hton/Ntoh | `pkg/im/cmd`、`pkg/im/header` |
| `lib/comm/const.go` | `pkg/im/const` |
| `lib/comm/key.go` | `pkg/rediskey` |
| `lib/comm/errno.go` | `pkg/errno` |
| `lib/comm/mesg.go` HttpResp | `pkg/httpx`（可选） |
| `doc/mesg/mesg.proto` + `lib/mesg/mesg.pb.go` | `proto/im/v1/mesg.proto` → `gen/go/im/v1` |
| Thrift seqsvr | `proto/seq/v1/seq.proto` → gRPC |

**迁移原则**：行为与必嗨 **二进制兼容**（48B 字段顺序、大端、CMD 数值不变），便于对照 beehive-im Demo 与 pcap。

| 包 | 用途 | 谁用 |

| `pkg/im/cmd` | `cmd.CMD_GROUP_CHAT` | 全部 L3 + hubclient 测试 |

## 5. IM 报头（48 字节）

与必嗨 `MesgHeader` **字段顺序、大小、大端** 一致：

| 偏移 | 字段 | 类型 | 说明 |
|------|------|------|------|
| 0 | cmd | uint32 BE | 命令 ID |
| 4 | length | uint32 BE | 报体长度（不含 48B 头） |
| 8 | sid | uint64 BE | 源会话 ID |
| 16 | cid | uint64 BE | 源连接 ID |
| 24 | nid | uint32 BE | 源/目的接入 NID（下行 async_send 为目的 nid） |
| 28 | seq | uint64 BE | 源流水号 |
| 36 | dsid | uint64 BE | 目标会话 ID |
| 44 | dseq | uint64 BE | 目标流水号 |

```go
// pkg/im/header/header.go（示意）
type Header struct {
    Cmd    uint32
    Length uint32
    Sid    uint64
    Cid    uint64
    Nid    uint32
    Seq    uint64
    Dsid   uint64
    Dseq   uint64
}

const Size = 48

func (h *Header) Marshal(buf []byte) error   // 大端写入
func Unmarshal(buf []byte) (*Header, error)
func (h *Header) Validate(flags ValidateFlags) error // nid/sid 规则
```

**校验规则**（迁移自 `IsValid`）：`nid != 0`；需要 sid 的场景 `sid != 0`。

**完整帧**：`header.Marshal + proto.Marshal(body)` → 作为 hubclient `AsyncSend` 的 payload（外层再由 core 包 20B bus wire）。

## 6. CMD 命令字

定义于 `pkg/im/cmd/cmd.go`，与必嗨 `lib/comm/mesg.go` **数值不变**。

### 6.1 分区（节选）

| 区间 | 含义 | 示例 |
|------|------|------|
| 0x01xx | 通用 / 生命周期 | `CMD_ONLINE` 0x0101、`CMD_PING` 0x0105 |
| 0x02xx | 私聊 | `CMD_CHAT` 0x0201 |
| 0x03xx | 群聊 | `CMD_GROUP_CHAT` 0x030B、`CMD_GROUP_JOIN` 0x0305 |
| 0x04xx | 聊天室 | `CMD_ROOM_CHAT` 0x040B、`CMD_ROOM_JOIN` 0x0405 |
| 0x05xx | 推送 | `CMD_BC` 0x0501 |
| 0x06xx | 内部上报 | `CMD_LSND_INFO` 0x0601 |

新增 cmd 流程：

1. 在 `cmd.go` 增加常量 + 注释  
2. 在 `proto/im/v1/mesg.proto` 增加 message  
3. **minor** semver 发版  
4. 使用该 cmd 的服务 `go get` 升级  

hubclient **只**在 SUB 配置里引用 cmd 值，不改库代码。

## 7. Protobuf（IM 消息体）

### 7.1 文件与版本

- 路径：`proto/im/v1/mesg.proto`
- `package hi.im.v1;`
- 语法：**proto3**（新栈）；从 beehive `doc/mesg/mesg.proto`（proto2）迁移时去掉 `required`，用 `optional` / 文档约定

生成：

```bash
protoc --go_out=gen/go --go_opt=paths=source_relative \
  proto/im/v1/mesg.proto
```

Go import：`github.com/<org>/hi-im-api/gen/go/im/v1`

### 7.2 cmd 与 message 映射

| CMD | message（必嗨名） |
|-----|-------------------|
| 0x0101 | mesg_online |
| 0x030B | mesg_group_chat |
| 0x040B | mesg_room_chat |
| … | 见 beehive `doc/mesg/mesg.proto` |

**M2 最小集**（生态冒烟）：`online`、`group_chat`、`group_join` + 对应 ACK。

## 8. Redis 键规范

包：`pkg/rediskey`

从必嗨 `lib/comm/key.go` 迁移，**字符串模板不变**，避免线上数据迁移。

| 宏（Go 常量） | 键模式 | 用途 |
|---------------|--------|------|
| `KeySidAttr` | `im:sid:%d:attr` | sid → uid/nid |
| `KeyGidToNidZSet` | `chat:gid:%d:to:nid:zset` | **群聊 fan-out 第一段** |
| `KeyGidToUidZSet` | `chat:gid:%d:to:uid:zset` | 群在线用户 |
| `KeyRidToNidZSet` | `chat:rid:%d:to:nid:zset` | 聊天室 fan-out |

提供函数式 API，避免手写 `fmt.Sprintf` 打错：

```go
func SidAttrKey(sid uint64) string
func GidToNidZSetKey(gid uint64) string
```

**变更 Redis key = breaking change**，仅 major semver。

### 9.1 errno

包：`pkg/errno` — 迁移 `lib/comm/errno.go`（`ERR_SYS_*`、`ERR_SVR_*`）。

```go
    Code   int    `json:"code"`

```

与必嗨 `HttpResp` 一致；**不含** Gin 中间件（留在各 L3 仓库）。

## 10. gRPC：seq/v1

替代必嗨 Thrift seqsvr；定义于 `proto/seq/v1/seq.proto`。

```protobuf
syntax = "proto3";
package hi.im.seq.v1;

option go_package = "github.com/<org>/hi-im-api/gen/go/seq/v1;seqv1";

service SeqService {
  rpc AllocSid(AllocSidRequest) returns (AllocSidResponse);
  rpc AllocRid(AllocRidRequest) returns (AllocRidResponse);
  rpc AllocGid(AllocGidRequest) returns (AllocGidResponse);
  rpc AllocSeq(AllocSeqRequest) returns (AllocSeqResponse);
}
```

- **hi-im-seqsvr**：实现 server  
- **hi-im-usrsvr / chatroom**：client  
- 生成：`protoc --go-grpc_out=...`

## 12. 版本与发版（semver）

| 变更类型 | 示例 | 版本 |
|----------|------|------|
| 新增 CMD + proto message | 房间礼物 | **minor** |
| 修复 header 编解码 bug | 兼容修复 | **patch** |
| 改 Redis key 字符串 | 键名变更 | **major** |
| 改 48B 布局 | 几乎不应发生 | **major** |

发版：

```text
git tag v0.2.0
```

```bash
go get github.com/<org>/hi-im-api@v0.2.0
```

**hi-im-hubclient** 应声明 `require hi-im-api v0.x` 下限，但不随每个 minor 强制升级（除非 hubclient 测试需要新 cmd）。

## 15. 风险与决策

| ID | 决策 | 理由 |
|----|------|------|
| A1 | 独立仓库，不合并 hubclient | api 常改、hubclient 求稳；见 beehive-im `doc/hi-im-档C技术方案设计.md` §4.5 |
| A2 | 48B/CMD 与必嗨兼容 | 对照迁移、Demo 复用 |
| A3 | proto3 新写，不沿用 proto2 | Go 生态与 gRPC 一致 |
| A4 | pkg/ 无 IO | 契约库零副作用 |
| A5 | 不叫 hi-im-sdk-go | 避免外部误用 |
